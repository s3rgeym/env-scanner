# Техническое задание

## 1. Назначение

Разработать консольную утилиту для массовой проверки набора веб-адресов на наличие публично доступных файлов и страниц, содержащих переменные окружения или диагностическую информацию, пригодную для анализа конфигурации приложения.

Утилита должна принимать список базовых URL, выполнять HTTP-запросы к заранее заданным путям, извлекать найденные переменные окружения и сохранять результат в машиночитаемом формате.

## 2. Цель разработки

Цель программы:

- автоматизировать проверку множества хостов на предмет открытых `.env`-файлов и диагностических PHP-страниц;
- сократить время ручной проверки;
- обеспечить воспроизводимый и формализованный формат результата;
- обеспечить безопасемое завершение, управляемую параллельность и предсказуемое поведение при ошибках.

## 3. Формат поставки

Результатом разработки должен быть CLI-инструмент на Go, запускаемый локально из исходного кода и в виде собранного бинарного файла.

В состав проекта должны входить:

- исходный код программы;
- модульные тесты;
- файл `docker-compose.yml` для локального тестового стенда;
- тестовые данные для проверки парсинга `.env` и `phpinfo`;
- документация по запуску и использованию.

## 4. Основной сценарий работы

Программа должна:

1. Получить на вход список базовых URL.
2. Для каждого URL сгенерировать набор проверяемых путей.
3. Выполнить HTTP GET-запросы к каждому пути.
4. Обработать успешные ответы со статусом `200 OK`.
5. Распознать тип содержимого по целевому пути.
6. Извлечь переменные окружения из ответа.
7. Сформировать результат в формате JSON Lines.
8. Записать результат в `stdout` или в указанный файл.

## 5. Проверяемые пути

Минимально обязательный набор путей:

- `/.env`
- `/.env.local`
- `/prod.env`
- `/dev.env`
- `/test.env`
- `/config/.env`
- `/docker/.env`
- `/phpinfo.php`
- `/info.php`

Требование к реализации:

- список путей должен быть централизован в коде;
- добавление новых путей должно требовать минимальных изменений;
- порядок обхода путей должен быть детерминированным.

## 6. Входные данные

Программа должна поддерживать чтение списка базовых URL:

- из `stdin`;
- из файла, путь к которому передан через аргумент командной строки.

Требования к формату входа:

- один URL или хост на строку;
- пустые строки должны игнорироваться;
- строки, начинающиеся с `#`, должны считаться комментариями;
- если схема не указана, программа должна автоматически подставлять `http://`.

Примеры допустимого входа:

```text
example.com
http://test.local
https://demo.example.org
```

## 7. Выходные данные

Формат выходных данных: JSON Lines, по одному JSON-объекту на найденный результат.

Каждый объект должен содержать:

- `timestamp` — время фиксации результата в UTC;
- `url` — полный URL проверенного ресурса;
- `response_time_ms` — длительность HTTP-запроса в миллисекундах;
- `env_vars` — объект `ключ -> значение` с извлечёнными переменными.

Пример:

```json
{"timestamp":"2026-03-25T04:28:33Z","url":"http://localhost:8080/.env","response_time_ms":1,"env_vars":{"APP_ENV":"production","DB_HOST":"localhost"}}
```

Требования:

- выход должен быть пригоден для последующей обработки `jq`, `grep`, `awk`, SIEM и др.;
- при записи в файл ошибки записи должны завершать программу с ненулевым кодом;
- частичная потеря результатов без возврата ошибки недопустима.

## 8. Логика обработки ответов

### 8.1. Для `.env`-подобных файлов

Необходимо поддержать разбор строк вида:

- `KEY=value`
- `KEY="value"`
- `KEY='value'`
- `export KEY=value`

Требования к парсеру:

- обрезать внешние одинарные и двойные кавычки;
- корректно обрабатывать значения с символом `=`;
- игнорировать пустые строки;
- игнорировать комментарии;
- удалять inline-комментарий только для некавыченных значений;
- сохранять исходные значения без дополнительной интерпретации escape-последовательностей.

### 8.2. Для `phpinfo.php` и `info.php`

Необходимо извлекать переменные из секции `Environment` HTML-страницы `phpinfo()`.

Требования:

- извлекать пары `name/value` из таблицы секции `Environment`;
- декодировать HTML entities;
- если секция `Environment` отсутствует, возвращать пустой набор переменных без падения программы;
- отсутствие данных в `Environment` не считается фатальной ошибкой запроса.

## 9. HTTP-поведение

Программа должна выполнять только `GET`-запросы.

Обязательные требования:

- на каждый запрос должен устанавливаться `User-Agent`;
- если кастомный `User-Agent` не задан, должен использоваться один случайно выбранный шаблон из заданного набора;
- должны поддерживаться таймаут TCP-соединения и общий таймаут HTTP-запроса;
- редиректы автоматически не переходятся;
- только статус `200 OK` считается успешным результатом для парсинга;
- тело ответа должно читаться с ограничением по размеру;
- ошибка чтения тела должна считаться ошибкой результата, а не игнорироваться.

Допустимо дополнительно выставлять служебные заголовки:

- `X-Forwarded-For`
- `X-Real-IP`

## 10. Параллельность и производительность

Программа должна поддерживать конкурентную обработку.

Требования:

- число воркеров задаётся через аргумент CLI;
- при `workers <= 0` программа обязана завершаться с ошибкой валидации без panic;
- должна поддерживаться опциональная глобальная rate limit-настройка в запросах в секунду;
- завершение должно быть корректным при отмене контекста или получении сигнала ОС;
- пайплайн `генератор задач -> воркеры -> writer` не должен терять данные и зависать.

## 11. Конфигурация CLI

Обязательные параметры командной строки:

- `--connect-timeout`, `-c`
- `--request-timeout`, `-r`
- `--workers`, `-w`
- `--rate-limit`, `-R`
- `--user-agent`, `--ua`
- `--log-level`, `-l`
- `--input`, `-i`
- `--output`, `-O`

Требования:

- все параметры должны иметь разумные значения по умолчанию;
- ошибки валидации конфигурации должны выводиться в `stderr`;
- при ошибке конфигурации программа должна завершаться с кодом `1`.

## 12. Логирование

Необходимо реализовать уровни логирования:

- `debug`
- `info`
- `warn`
- `error`

Требования:

- логирование должно быть потокобезопасным;
- сообщения пишутся в `stderr`;
- уровень логирования должен фильтроваться централизованно;
- при `debug` должны быть видны этапы генерации задач, выполнения запросов и записи результатов.

## 13. Обработка ошибок

Фатальными ошибками считаются:

- неверная конфигурация запуска;
- невозможность открыть входной файл;
- невозможность открыть файл результата;
- ошибка записи результата;
- ошибка чтения тела HTTP-ответа;
- неустранимые ошибки инфраструктуры, мешающие продолжить работу.

Нефатальными считаются:

- `404/403/500` по конкретному пути;
- сетевые ошибки по отдельному хосту;
- отсутствие секции `Environment` в `phpinfo`;
- пустой набор переменных в ответе.

Требования:

- единичная ошибка одного URL не должна ронять весь процесс, если writer и конфигурация корректны;
- фатальная ошибка writer должна завершать весь запуск с ненулевым кодом;
- panic в штатных сценариях эксплуатации недопустим.

## 14. Тестирование

Обязательно предусмотреть:

- unit-тесты парсера `.env`;
- unit-тесты генерации User-Agent;
- unit-тесты валидации конфигурации;
- unit-тесты обработки ошибок чтения тела ответа;
- unit-тесты возврата ошибки writer;
- unit-тесты генерации списка задач по всем путям;
- прогон `go test ./...`;
- прогон `go test -race ./...`;
- прогон `go vet ./...`.

## 15. Интеграционное тестирование

В проекте должен быть локальный тестовый стенд на `docker compose`.

Состав стенда:

- `nginx`;
- `php-fpm`;
- тестовые файлы в каталоге `testdata/www`.

Нужно обеспечить проверку:

- успешной отдачи `/.env`;
- успешной отдачи `/phpinfo.php`;
- ожидаемого отсутствия `/info.php`, если файл не создан;
- корректного состава полей `env_vars` в выходном JSON.

## 16. Нефункциональные требования

Требования к качеству реализации:

- язык разработки: Go;
- код должен быть читаемым и разбитым по понятным зонам ответственности;
- все изменения должны форматироваться через `gofmt`;
- программа должна корректно работать в Linux;
- по возможности должна работать без внешних зависимостей, кроме стандартной библиотеки и минимально необходимых Go-модулей.

## 17. Ограничения и допущения

- программа предназначена для использования только в разрешённых сценариях аудита, тестирования или внутренней диагностики;
- программа не должна сама обходить аутентификацию, исполнять payload или использовать техники эксплуатации;
- программа только проверяет доступность и извлекает уже публично отданное содержимое;
- HTTPS-проверка выполняется стандартным HTTP-клиентом Go с системной валидацией сертификатов;
- поддержка прокси, cookie jar и headless browser не входит в базовый объём работ.

## 18. Критерии приёмки

Работа считается выполненной, если:

- программа принимает список URL из файла и `stdin`;
- программа проверяет все заданные пути;
- `.env` и `phpinfo` корректно распознаются и парсятся;
- JSON Lines формируется без нарушения структуры;
- ошибки записи результата не теряются;
- невалидный `workers` не приводит к panic;
- unit- и интеграционные проверки проходят;
- запуск против локального `docker compose` стенда даёт ожидаемые результаты.

## 19. Возможные расширения

Не входят в обязательный объём, но допускаются как следующий этап:

- конфигурируемый список путей через CLI или YAML;
- экспорт в CSV;
- дедупликация результатов по хосту и пути;
- добавление HTML-отчёта;
- фильтрация по статус-кодам;
- поддержка дополнительного набора диагностических файлов;
- ограничение по доменам, подсетям и спискам исключений.
